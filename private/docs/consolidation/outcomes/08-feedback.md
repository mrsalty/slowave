# 08 — Feedback: Outcome Notes

Full plan: `plans/08-feedback.md`. Core doc: `core/08-feedback.md`. All 8 phases done.

## Root Problem Found

The generated `08-feedback.md` was internally consistent but **untested against the implementation** in three ways the audit surfaced:

1. It claimed `needs_review` is set "when `review_pressure ≥ stale_review_threshold` or `wrong_review_threshold`" — no such comparison exists anywhere in `slowave/core/services/feedback.py`. `needs_review=True` is unconditional given `apply_stale_wrong_review`; `stale_review_threshold`/`wrong_review_threshold` only set the *value* stored in the signal's `review_pressure` field for analytics.
2. It documented `useful_confidence_delta` (`+0.05` in the old doc, `+0.02` in code) as an active parameter. `SchemaStore.reinforce()` — the only call path `"useful"` feedback routes through — has no confidence parameter at all. The value is computed into `FeedbackSignal.confidence_delta` and then dropped; no schema's confidence is ever touched by this field, at any value.
3. It was missing 6 of 29 `FeedbackConfig` fields (`recall_feedback_weight`, `min_salience`, `min_confidence`, `max_confidence`, `missing_creates_memory`, `missing_replay_enabled`) and every template section past "Key Invariants" (Data Flow, Implementation Files, Diagnostic Hooks, Parameter Sensitivity, Known Failure Modes, Relationship to Other Modules).

Doc rewritten `core/08-feedback.md` 131 → ~300 lines, all template sections present, all 29 config fields listed and cross-checked against the dataclass, formulas spot-checked against the real `reinforce()`/`adjust_feedback_state()`/`refresh_utility()` implementations.

## A Structural Difference From Every Prior Module

This is the first module where **no existing project benchmark can measure it at all**. `locomo_eval.py`, `longmemeval_eval.py`, `stalememory_eval.py`, and the wiki/dmr/temporal_eval scripts never call `retrieval_feedback()`/`record_retrieval()` — grep confirms zero hits for `feedback`/`reinforce` across `tests/integration/*.py`. This isn't a benchmark bug: these are single-shot offline recall evals, and feedback requires a multi-turn interaction with an explicit quality label that none of them simulate. `iteration-strategy.md` predicted exactly this back when it prioritized this module 6th of 8 ("Requires many sessions to accumulate signal. Can't measure efficacy in a single benchmark run.").

Two substitute evidence sources filled the gap:

**Live dogfood DB** (`~/.slowave/slowave.db`, 47 schemas, 30 real `context_feedback_events`, 55 `context_recall_events` — accumulated from this project's own Claude Code sessions calling `slowave_activate`/`slowave_reinforce`, not synthetic). This is possible because `slowave_reinforce` — the MCP tool this session itself is required to call at close — is the direct public entry point for `FeedbackService.retrieval_feedback()`; the module under study is the same mechanism backing this project's own long-term-memory tooling.

**A new deterministic ablation script** (`scripts/feedback_ablation.py`) against a real `SlowaveEngine`, substituting for the "run an eval script with a flag on/off" pattern every prior module used, since no eval script here has anything to toggle against.

## Findings, Confirmed Two Ways (Live Data + Controlled Script)

**The noise-score demotion rule is alive on real data, but softer than its own code comment claimed.** Live DB: 6/47 schemas have 6-8 cumulative `irrelevant` marks, 0 `used` marks, `context_noise_score` ∈ [0.857, 0.889], boolean `needs_review=1` — and **`status="active"`** in every case. Only 1/47 schemas (marked `wrong` with `outcome="failure"`) has `status="needs_review"` (the string). Reproduced exactly in the controlled script: after 4 `irrelevant` marks with a `scope_id` set, the boolean flag fires and the schema remains fully returned by `eng.context()`; a `wrong`+`outcome="failure"` schema's `status` string is escalated and it *is* excluded. The pre-existing code comment ("Demotion: consistently-negative memories leave default context entirely") overstated this — fixed to describe the soft-penalty-only reality (comment-only change, `schema_store.py`, no behavior change, same bar as the Temporal module's docstring fix).

**A new discovery made *while building* the reproduction, not anticipated by the original diagnostic questions**: the noise-counting query (`_update_utility_scores`, `schema_store.py:830-840`) filters `WHERE scope_id IS NOT NULL`. The first version of the ablation script called feedback with no `scope_id` and got `context_noise_score=0.0` — the demotion rule silently never fires for feedback events that carry no scope. Re-running with `scope_id` set reproduced the live-DB numbers exactly. This is now Invariant 10 in the core doc and a new Known Failure Mode row.

**Three config fields are genuinely dead** (`apply_outcome_to_schema_reward`, `missing_creates_memory`, `missing_replay_enabled`) — confirmed by grep (zero read sites outside their dataclass definition) and by the ablation script (identical schema/DB state regardless of value). Not fixed — building the missing behavior (outcome-driven schema reward, missing-triggered memory creation/replay) is a product decision this investigation didn't make on its own, so these are left as explicitly-flagged open items rather than blind implementations.

**Two threshold fields are cosmetic** (`stale_review_threshold`, `wrong_review_threshold`) — confirmed by code read (no comparison exists) and by the script (`needs_review=True` identically at 0.0, the default, and 999.0). Not fixed: `review_pressure` is currently defined to *equal* the threshold by construction for the label it belongs to, so even adding a `review_pressure >= threshold` gate would be a tautology — a real fix would need `review_pressure` computed from an independent signal (e.g. cumulative negative-mark count), which is a design change, not a bug fix.

**Salience bounds are asymmetric by construction, and this is real, not a documentation gap.** The `"useful"` path (`SchemaStore.reinforce()`) has a ceiling (`20.0`) and no floor. The `partially_useful`/`irrelevant`/`stale`/`wrong` path (`SchemaStore.adjust_feedback_state()`) has a floor (`min_salience`, default `0.01`) and no ceiling. Verified by the ablation script: 500 `partially_useful` reps at full (`recall`) weight push salience to `20.9999999999997` (≈1.0 + 500×0.04), past where the `useful` path would have saturated at exactly `20.0`. Documented as Invariant 5 and a Parameter Sensitivity row; not changed, since there's no evidence this asymmetry causes an observed problem (no benchmark or live-data signal of runaway `partially_useful` accumulation).

## Design Evaluation

Qualitative assessment of the module's design, beyond the doc-accuracy and ablation findings above.

**Strong points:**
- The `FeedbackSignal` decomposition is genuinely well-conceived: separating *why* a memory failed (`context_fit`, `truth_error`, `temporal_error`, `missingness`, `overload`) from *what changes as a result* (`salience_delta`, `confidence_delta`) is a clean design that lets `missing`/`too_much_context` carry real diagnostic signal without ever risking a schema mutation — the two categories of label are handled by the same data structure without special-casing the mutation path.
- Outcome-vs-memory-quality separation (Invariant 2) is architecturally sound, not just a quirk. `apply_outcome_to_schema_reward` defaulting `False` is a deliberate choice to stop task-level noise (an agent that fails despite good memories, or succeeds despite bad ones) from corrupting the memory-quality reward channel. This is the right call for a system meant to accumulate signal over many unrelated sessions.
- Source-weighting (`context_feedback_weight=0.5` vs `recall_feedback_weight=1.0`) encodes a sensible intuition — an explicit `slowave_reinforce` call on memories a caller says it relied on (`"recall"`) is stronger evidence than passive context-injection exposure (`"context"`) — and the live dogfood DB shows this distinction is exercised in practice (28:2 in favor of the weaker `"context"` path), not just theoretical.
- Auto-derivation (`_derive_context_fields`) and graceful degradation (synthesizing a minimal parent row when no snapshot exists) both reduce caller burden and avoid silently dropping signal. A caller only has to supply the label and the id lists; the module fills in the rest and never refuses to record feedback for lack of a prior snapshot.

**Where the design is weaker than it looks:**
1. **The 10-component signal vector implies more behavioral richness than the code delivers.** `review_pressure` is computed and stored on every `stale`/`wrong` event but gates nothing; `useful_confidence_delta` is computed but unreachable from the only call path that could use it; `outcome_reward` is computed and stored but never applied regardless of `apply_outcome_to_schema_reward`'s value. That's 3 of 10 signal components, plus 2 of the 29 config fields that exist specifically to tune one of them, with zero behavioral effect. The data model reads as neuroscience-inspired and comprehensive; the actual behavioral surface is narrower than the schema suggests, and nothing in the code or the old doc flagged the gap until this investigation measured it.
2. **Two similarly-named fields mean genuinely different things, and the source code's own comment fell into the trap.** Boolean `needs_review=1` (soft ranking penalty via `context_noise_score`) and string `status="needs_review"` (hard exclusion from default retrieval) share a name and a superficial purpose ("flag this for attention") but have opposite-in-kind effects on retrieval. The `_update_utility_scores` docstring comment describing the boolean path claimed the stronger (string-path) behavior before this investigation's fix — a naming collision that actively misled a future reader, not merely an omission.
3. **The salience-bound asymmetry (ceiling on `reinforce`, floor-only on `adjust_feedback_state`) shows no sign of being a deliberate, documented tradeoff.** `reinforce()`'s 20.0 cap has an explicit code comment justifying it (ranking normalizes by /20, growth past that is invisible); `adjust_feedback_state()` has no equivalent comment or ceiling parameter at all. This reads like two methods written for two different original purposes (recall-time reinforcement vs. feedback-driven adjustment) that were later both pressed into feedback service, rather than one bound design applied consistently.
4. **The `scope_id` precondition on noise tracking is an unannounced footgun.** Nothing in the `FeedbackConfig` docstring, the MCP tool description, or the old core doc mentioned that omitting `scope_id` makes `context_noise_score`/demotion permanently inert for that feedback stream. A caller doing everything else right (correct labels, correct ids, correct outcome) gets silently zero self-cleaning if scope is absent — discoverable only by reading `_update_utility_scores`'s SQL or, as happened here, by building a reproduction script and getting a surprising null result.
5. **Most fundamentally: a module whose entire purpose is closing the loop between usage and memory quality has never been run through a loop that actually closes.** Every numeric constant in this module — 7 salience/confidence deltas, 2 review thresholds, 2 source weights — was set by design intuition, not measurement, and none of the 6 project benchmarks can validate or refute them because none simulate the multi-turn, labeled interaction the module requires. The live dogfood DB is real evidence that the mechanism *fires*, but 30 events is far too few to say whether the specific deltas are well-calibrated. This isn't a flaw in this investigation's rigor — it's a real, load-bearing gap in the project's evaluation infrastructure that happens to be invisible until you look at this specific module.

**Bottom line:** the core idea — decompose symbolic feedback into an interpretable numeric vector, then let salience/confidence drift accumulate slowly over many sessions — is sound and matches how the rest of the system treats salience elsewhere (small monotonic deltas, no single event dominating). But the implementation has visibly outpaced its own specification in three ways: dead signal components that look load-bearing, an undocumented precondition (`scope_id`) that silently disables self-cleaning, and two same-named-different-meaning state fields that already fooled the module's own source comment. None of these are benchmark-measurable regressions — they're latent correctness risks that only a real, feedback-exercising benchmark (which does not exist yet) or careful code reading (as done here) can surface. Trust the architecture's shape; do not assume the specific constants or the "richness" of the signal vector reflect what actually happens to a schema.

## Follow-up (2026-07-10): acting on the Design Evaluation

The user asked, item by item, what to do about the 5 Design Evaluation weaknesses above. Decisions and what was actually done:

**1. The 3 dead signal components — remove or plug in? Answer: mixed, decided per-field.**
- `useful_confidence_delta`: **plugged in.** `SchemaStore.reinforce()` gained an optional `confidence_delta`/`min_confidence`/`max_confidence`, clamped the same way `adjust_feedback_state()` already was; `FeedbackService`'s `useful` branch now passes `useful_signal.confidence_delta * source_weight`. Verified: from a starting confidence of 0.5, a `useful` call with `useful_confidence_delta=0.2` (default `context` weight 0.5) moves confidence to exactly 0.6; from confidence 1.0 (already at ceiling), a `useful_confidence_delta=0.9` call clamps at 1.0 rather than overshooting.
- `stale_review_threshold`/`wrong_review_threshold`: **removed** — `review_pressure` is defined to equal the threshold by construction, so a real gate would be tautological without redefining `review_pressure` from an independent signal (a design change, not a bug fix). Replaced with fixed module constants `_STALE_REVIEW_PRESSURE=0.7`/`_WRONG_REVIEW_PRESSURE=1.0` that reproduce the exact same stored values for default configs.
- `apply_outcome_to_schema_reward`/`missing_creates_memory`/`missing_replay_enabled`: **removed**, per explicit instruction — these represent unbuilt features, not bugs; whether to build the underlying behavior (outcome-driven schema reward, missing-triggered memory creation/replay) remains an open product decision, deliberately not made here.

**2. The `needs_review`/`status="needs_review"` naming collision — explained further, not yet changed.** Gave a concrete trace using the exact live-DB numbers from the original investigation (schemas 21-26: boolean flag + `status="active"`, still retrievable; schema 10: `wrong`+`failure` → `status` string escalated, excluded). Three fix options were presented (clarify via helper properties / rename the column via migration / unify the two mechanisms behaviorally) — **no option chosen yet**, left as an explicit open item pending a decision.

**3. Salience-bound asymmetry — explained, then fixed per explicit instruction.** `SchemaStore.adjust_feedback_state()` gained a `max_salience` parameter defaulting to a new shared module constant, `SchemaStore.SALIENCE_CEILING = 20.0` (also now used by `reinforce()`, replacing its hardcoded literal), with a comment explaining the fix's motivation directly in the code. Verified: 500 `partially_useful` reps (which would reach 21.0 uncapped) now saturate at exactly 20.0, matching `reinforce()`'s behavior.

**4. `scope_id` footgun — documented + warned per explicit instruction; sentinel-fallback documented but not implemented.** Added: a paragraph to `retrieval_feedback()`'s docstring, a note on the `slowave_reinforce` MCP tool description, and a `logging.WARNING` (logger `slowave.core.services.feedback`) fired whenever `scope_id` is unavailable and `irrelevant`/`stale`/`wrong` ids are given (verified silent for `useful`-only calls, which don't need `scope_id`). The third option discussed — defaulting to a sentinel scope (e.g. `"unscoped"`) so tracking always accumulates *somewhere* instead of just becoming visible-but-still-absent — was **not implemented**: it changes real behavior (pooling otherwise-unrelated callers' feedback into one bucket) rather than just adding visibility, and was flagged as a separate decision rather than bundled into this fix.

**5. Measurement — `scripts/feedback_ablation.py` rewritten into a scored micro-benchmark, per explicit instruction.** Replaced the original ad-hoc F1-F7 pass/fail script (superseded by `tests/unit/test_feedback_review_gating.py`, which now formalizes those findings as regression tests) with a deterministic, scored internal-consistency benchmark: synthetic "good"/"bad" schemas receive noisy (20% mislabeled), repeated `recall→feedback` rounds over multiple simulated sessions, and the script reports an AUC-style separation score (P(good schema's score > bad schema's score), 1.0 = perfect, 0.5 = chance) for both final salience and `context_noise_score`, plus `status`-exclusion and ceiling-saturation rates. Three configs are compared in one run — baseline, `apply_learning=False`, `scope_id=None` — with baked-in sanity assertions (non-zero exit code on inversion).

**Results (`n_good=12, n_bad=12, rounds=20, seed=42`, all 5 sanity checks pass):**

| Config | `salience_auc` | `noise_score_auc` | `bad_excluded_%` |
|---|---|---|---|
| baseline | **1.0** | **1.0** | 66.7% |
| `apply_learning=False` | 0.5 (collapsed) | **1.0** (unaffected!) | 0.0% |
| `scope_id=None` | 1.0 (unaffected) | 0.5 (collapsed) | 66.7% |

**New finding, discovered while building this benchmark (not anticipated by the prior investigation):** `apply_learning=False` does **not** disable the noise-score/`needs_review` demotion mechanism — only the direct salience/confidence/status mutations. `FeedbackService.retrieval_feedback()` calls `schemas.refresh_utility()` for every touched id **unconditionally**, outside the `if self.cfg.apply_learning:` block, because `context_noise_score` is recomputed directly from the always-persisted `context_feedback_events` rows regardless of whether direct reinforcement ran. The `apply_learning=False` row above proves this directly: `noise_score_auc` stays at a perfect 1.0 even though `salience_auc` collapses to chance. This means "the master learning gate" is a narrower gate than its name and its own doc previously implied — it does not, in fact, disable all feedback-driven learning, only the *direct* channel. Locked in as a new core doc invariant (11) and a new regression test (`test_apply_learning_false_does_not_disable_noise_score_demotion`). **Not changed** — whether `apply_learning` *should* also gate the derived noise mechanism is a design question flagged for a future decision, not treated as an obvious bug given `apply_learning=False` was never previously verified against this specific mechanism (the original F1 ablation test used no `scope_id`, which masked this entirely).

All fixes: 43 tests pass (`test_context_feedback.py` + `test_feedback_review_gating.py`, up from 26+16=42 due to the new scope_id-warning and apply_learning-nuance tests), full unit suite unaffected, black/isort/mypy clean.

## Follow-up (2026-07-10, part 2): the Labile State & Reconsolidation

Part 1 (above) left Design Evaluation weakness #2 (the `needs_review`/`status="needs_review"` naming collision) explicitly undecided, and treated the `scope_id` gap (weakness #4) as a low-priority footgun since the flag it fed was, at the time, a dead end regardless of `scope_id`. Both conclusions changed after a deeper design conversation, prompted by the user sharing dashboard screenshots that made the naming collision concrete rather than theoretical.

### How the redesign was reached

The dashboard showed two symptoms that trace back to exactly the same root cause: a top-level "1 warning: 1 schemas need review" banner that only counts `status="needs_review"` (the string, 1/47 live schemas), sitting above a table where nearly every row carries a "⚠ review" badge that actually renders the *boolean* `needs_review` column (6+/47 live schemas) — two different underlying fields, rendered with overlapping English, counted by two different queries. Tracing further: **no code anywhere ever clears the boolean flag once set, and nothing ever acts on it except a soft ranking penalty in one retrieval path and a much harsher (80%, `× 0.20`) score multiplier in the other** (`services/retrieval.py`'s `recall()` — a mechanism this investigation hadn't previously documented). Three separate subsystems (`decay_unused()`, `remember()`'s ambiguous-update case, feedback's noise-demotion rule) all flag a schema "for later," using language that implies a follow-up step (decay's docstring literally says "flagged for eventual pruning") — and none of the three ever built that follow-up. Consolidation, the most plausible place for it, was checked directly and confirmed to never read the boolean at all.

Reconstructing this against real memory science (the user's explicit ask, given Slowave's brain-inspired framing) landed on standard reconsolidation-theory terminology, not an invented metaphor: a reactivated memory that doesn't cleanly match expectations enters a temporarily unstable state called **labile** — open to revision, not yet resolved. The process by which a labile trace resolves (restabilizing back to what it was, or being updated/replaced by better evidence) is called **reconsolidation**. This maps cleanly onto the codebase's own existing vocabulary: "consolidate" already means "build new schemas from fresh replay"; "reconsolidate" is its natural sibling, "revisit ones already flagged as uncertain." (An earlier, cheaper naming attempt — `resolve_labile_schemas()` / `"labile_resolution"` — paired the correct state word with a generic, unrelated process noun; the fix was pairing "labile" with "reconsolidation" instead, not replacing "labile" with something else.)

**Scoping decision, made explicitly and not yet reversed:** the `needs_review` **column name itself** was **not** renamed to `is_labile`. Doing so is a real migration (schema.sql, every Python read/write site, the CLI, and — critically — the dashboard's JS/HTML, a different tech stack outside everything else touched in this investigation) and was deliberately left as a separate, future decision. "Labile"/"reconsolidation" are the vocabulary used in code comments, docstrings, and this document; the database column is still literally named `needs_review`.

### Three things were built, all reusing existing machinery rather than inventing new mechanisms

**1. Consolidation reconsolidates labile schemas** (new: `Consolidator.reconsolidate_labile_schemas()`, `slowave/core/consolidation.py`, wired into `ConsolidationService.consolidate_once()`, `slowave/core/services/consolidation.py`). Brain analogue: hippocampal replay reactivates a labile trace alongside related memories during consolidation; if nothing challenges it, it restabilizes, if it's contradicted or cleanly superseded by better/more-recent evidence, it resolves that way instead. Mechanically: pulls up to 20 `status="active"` schemas with `needs_review=True` per pass, finds each one's nearest active neighbor (`related_schema_cosine=0.72`, the same threshold `_best_related_schema` already uses), and replays it through the *same* `GeometricContradictionJudge` and the *same* caution gates (`min_support_to_supersede`, `min_time_delta_to_supersede_s`) the fresh-schema consolidation path already uses — no new judge, no new gates, no new thresholds to calibrate.

   One correctness subtlety worth recording since it wasn't obvious at first: `GeometricContradictionJudge.judge()`'s `old=`/`new=` argument slots are **not symmetric** — on a `"contradicts"` verdict, the existing calling convention (`_write_latent_schema`) always demotes whichever schema was passed as `old`, regardless of the verdict's `time_delta_s` sign (that sign only picks the `"supersedes"` vs `"contradicts"` *label*). Always passing the labile schema as `new` (the "winning" slot) would have silently biased every contradiction in its favor purely by construction — the schema already under suspicion would always beat a neutral neighbor. The fix: decide which schema goes in which slot by actual chronology (`last_updated_ts`, since consolidation-authored `mean_ts` isn't populated on non-consolidation schemas) *before* calling the judge, so the more recent evidence wins on its merits, not its argument position.

   Outcomes per labile schema, each covered by a test in `tests/unit/test_labile_lifecycle.py`: **restabilized** (no conflict found — `needs_review` cleared), **superseded**/**contradicted** (neighbor demoted, labile schema confirmed as the winner, `needs_review` cleared either way), or **inconclusive** (no sufficiently related neighbor exists — left labile; this is the brain analogue of passive extinction, a trace nothing challenges or corroborates just keeps decaying via `decay_unused()` rather than being actively resolved).

**2. Feedback clears lability directly.** An explicit `useful`/`partially_useful` mark is direct positive evidence a flagged schema is still good — it now clears `needs_review` immediately rather than waiting for the next consolidation pass. `SchemaStore.reinforce()` (the `useful` path) gained a `clear_needs_review` parameter; `adjust_feedback_state()` (the `partially_useful` path) already supported a `needs_review` parameter and now gets `needs_review=False` passed explicitly. `irrelevant`/`stale`/`wrong` do not clear it (verified by test).

**3. Sustained recurrence clears lability passively.** If a labile schema keeps getting genuinely reactivated afterward (`SchemaStore.reinforce()` called with `recall_hit=True`, its default), that reactivation is itself evidence the memory is still good — mirroring how repeated recall drives real memory consolidation. `_update_utility_scores` now tracks a `recurrence_count_at_flag` facet, captured lazily the first time it observes an already-flagged schema with no baseline recorded yet, and clears `needs_review` once `_RECONSOLIDATION_RECOVERY_RECURRENCE` (3) recurrence hits accumulate *since* that baseline — deliberately not lifetime recurrence, so a schema that already had 10 recurrences before ever being flagged doesn't recover the instant it's flagged (`test_recurrence_before_flag_does_not_count_toward_recovery`). On the rare same-call conflict where a schema simultaneously qualifies for the noise-based demotion *and* has crossed the recovery threshold, demotion wins — explicit accumulated negative feedback is stronger, more deliberate evidence than passive reactivation.

### The `scope_id` fix, revisited

Weakness #4's priority was reassessed once `needs_review` stopped being a dead end: a memory that should be entering the reconsolidation pipeline but silently never gets flagged (because the feedback that would have flagged it carried no `scope_id`) now means it never gets replayed, never gets a chance to restabilize or be properly superseded — a bigger miss than a lost ranking nudge. The fix itself is unchanged from what was scoped in part 1's discussion: `_update_utility_scores`'s counting query (`schema_store.py`) no longer filters `WHERE scope_id IS NOT NULL` — that filter only ever gated on non-null-ness, never on the scope *value*, so removing it has no effect on the separate, still-independently-filtered cross-scope generalization computation. The now-inaccurate warning (`services/feedback.py`, "this won't count toward context_noise_score") was removed rather than left describing a gap that no longer exists; docstrings and the MCP tool description were updated to match.

Before implementing, the live DB and the oldest available backup (spanning 2026-06-24 to 2026-07-06, before the current live DB's window) were checked for how often this had actually mattered: **zero confirmed real instances**, across this project's entire history, of a negative mark on an actual memory being lost to this gap. The precondition (no `scope_id`) occurred at a real, non-trivial rate historically (~6% of calls) but never once coincided with a feedback call that actually named a target schema. This reframed the fix from "there's live damage happening" to "a latent robustness gap worth closing now that it feeds something real, but not evidence of a live incident" — which is why it was bundled into this same pass rather than treated as an emergency on its own.
